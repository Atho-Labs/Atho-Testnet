ATHO BLOCK STORAGE REFERENCE
BITCOIN-STYLE RAW BLOCK FILES + LMDB FOR EVERYTHING ELSE
INCLUDING BLOCK METADATA IN LMDB
========================================================

Reference status
----------------
This document is a reference for the Atho implementation.

It covers block storage architecture only.

Atho is keeping:
- full raw block storage in Bitcoin-style flat block data files
- everything else in LMDB

That means:
- raw full blocks -> flat block files
- block metadata -> LMDB
- block index -> LMDB
- UTXO set -> LMDB
- tx index -> LMDB
- chain metadata -> LMDB
- peer metadata -> LMDB
- mempool metadata if persisted -> LMDB
- validation metadata -> LMDB

This is the recommended long-term production structure because it gives Atho:
- fast append-only block storage
- fast indexed lookups
- fast live-state access
- clean architecture
- easier recovery and reindex
- better long-term scaling
- a Bitcoin-proven storage pattern

============================================================
1. SHORT VERSION
============================================================

Atho should use a hybrid storage model:

1. Store full raw blocks in flat files, Bitcoin-style
2. Store all indexes, metadata, and live state in LMDB

That includes block metadata.

This is important:
LMDB should not only store pointers to the block location.
LMDB should also store the important metadata about the block so the node does not need to reopen and reparse raw block files for every normal question.

============================================================
2. WHAT THIS DOCUMENT IS AND IS NOT
============================================================

This document is only about block storage architecture.

This document is NOT changing:
- UTXO storage strategy
- chainstate strategy
- tx index strategy
- mempool state strategy
- peer database strategy
- general LMDB strategy for mutable state

Those all stay in LMDB.

The change recommended here is:
- move full raw block blobs into flat block files
- keep all block metadata and indexes in LMDB

============================================================
3. THE BITCOIN MODEL
============================================================

Bitcoin uses a hybrid design.

Bitcoin stores:

A. Full raw blocks
- in flat files named like:
  - blk00000.dat
  - blk00001.dat
  - blk00002.dat

B. Block index and chain metadata
- in a database

This means Bitcoin does NOT use the flat block file as the live searchable state database.
It uses the flat block file as raw archival storage.

This is the main model Atho should follow.

============================================================
4. BITCOIN BLOCK FILE STRUCTURE
============================================================

Each block in Bitcoin's blk*.dat files is stored as:

    [network magic][block length][raw serialized block]

Field breakdown:
----------------

1. Network magic
   - 4 bytes
   - identifies the network
   - helps locate block boundaries
   - helps reject wrong-network parsing

2. Block length
   - 4 bytes
   - little-endian unsigned integer
   - length of the raw serialized block that follows

3. Raw serialized block
   - full block bytes in canonical consensus format
   - includes:
     - block header
     - transaction count
     - all serialized transactions
     - witness-aware transaction structure where applicable

This is intentionally simple.
That simplicity is one reason it scales well.

============================================================
5. WHAT THE RAW BLOCK ITSELF CONTAINS
============================================================

Inside the raw serialized block, the structure is basically:

    [block header][tx_count][transactions...]

The block header contains:
- version
- previous block hash
- merkle root
- timestamp
- bits / difficulty encoding
- nonce

Then:
- CompactSize transaction count
- full serialized transactions

If witness-aware serialization is used, the transactions include witness-related serialization as defined by consensus.

The raw block file is not a rich query database.
It is raw archival block storage.

============================================================
6. WHY BITCOIN STORES BLOCKS THIS WAY
============================================================

Bitcoin-style flat block files are a good fit because blocks are:

- large binary blobs
- append-only
- historical
- rarely modified after write
- often replayed sequentially
- naturally suited to file offsets

Benefits:
- fast sequential append
- low write overhead
- low storage overhead
- simple archival format
- easy to back up
- easy to replay for reindex
- easy to rebuild indexes from raw history

============================================================
7. WHY BITCOIN DOES NOT STORE UTXOS IN BLOCK FILES
============================================================

UTXOs are a completely different workload.

UTXO state is:
- live
- constantly changing
- randomly accessed
- randomly inserted/deleted
- part of the hot validation path
- not append-only archival data

A flat file is bad for UTXOs because:
- random lookup is awkward
- update/delete is awkward
- indexing becomes fragile
- hot-state access becomes slower and more complex

So the lesson is:

- immutable historical blocks -> flat files
- mutable hot state -> database

Atho should do the same.

============================================================
8. ATHO RECOMMENDED HYBRID MODEL
============================================================

Atho should use:

A. Flat block files for full raw blocks
B. LMDB for block metadata, indexes, and all live state

This is the correct split.

Flat files:
- store the canonical raw serialized blocks
- append-only
- historical source of truth for full block bytes

LMDB:
- stores where blocks are
- stores what the blocks are
- stores chain relationships and validation state
- stores all hot indexed state

============================================================
9. EXACT RECOMMENDATION FOR ATHO
============================================================

Use Bitcoin-style block data files for raw blocks.

Suggested naming:
- blk00000.dat
- blk00001.dat
- blk00002.dat

Suggested network-specific paths:
- data/mainnet/blocks/
- data/testnet/blocks/
- data/regnet/blocks/
- data/sandbox/blocks/

Recommended per-block stored structure:
---------------------------------------

    [network magic][block length][raw serialized block]

Optional storage-only helper:
- file checksum or per-block checksum if desired for storage integrity
- but keep storage helper fields separate from consensus serialization

============================================================
10. WHAT STAYS IN LMDB
============================================================

Atho is keeping everything important for active lookup/state in LMDB.

Recommended LMDB responsibilities:

1. Block index
2. Block metadata
3. Chainstate / UTXO set
4. Transaction index
5. Peer database
6. Mempool metadata if persisted
7. Validation metadata
8. Chain tip metadata
9. Reorg / side-chain / status metadata

This is not pointless duplication.
This is the correct use of LMDB in the hybrid model.

============================================================
11. BLOCK METADATA SHOULD ABSOLUTELY STAY IN LMDB
============================================================

This is important.

Even if the full raw block bytes live in blk*.dat files, LMDB should absolutely keep block metadata.

That is not wasteful.
That is the purpose of the hybrid design.

Without metadata in LMDB, the node would have to:
- reopen block files frequently
- seek around more often
- reparse blocks repeatedly
- rebuild normal answers repeatedly

That would be slower and messier.

LMDB should answer normal questions like:
- what height is this block?
- what file is it in?
- what offset is it stored at?
- how long is it?
- what is its parent?
- what is its timestamp?
- is it valid?
- is it on the main chain?
- is it on a side branch?
- what is its chainwork?
- what is its tx count?
- what are its status flags?
- what is the current best tip?

That is exactly why block metadata belongs in LMDB.

============================================================
12. RECOMMENDED BLOCK METADATA IN LMDB
============================================================

LMDB should store block metadata such as:

Core identity:
- block hash
- height
- previous block hash
- header hash if separated conceptually
- network id

Storage location:
- file number
- byte offset
- byte length
- raw size
- vbyte/weight metadata if useful
- storage status flags
- pruning status if ever used

Consensus/header metadata:
- version
- merkle root
- witness root if used
- timestamp
- bits / target encoding
- nonce
- difficulty / target metadata if cached
- chainwork
- ruleset/version activation context if relevant

Validation and chain status:
- validation status
- fully validated flag
- header-only validated flag
- main-chain / side-chain / orphan status
- best-chain eligibility metadata
- failure/rejection status if tracked
- received time / processed time if useful
- block tx count
- coinbase txid if useful
- file commit status / persisted status if useful

Optional operational metadata:
- arrival source or peer source if intentionally stored
- compact-block reconstruction metadata if appropriate
- benchmark/debug timing metadata in sandbox only if desired
- pruning/archive markers if future features need them

The exact schema may vary, but the principle is:
LMDB should contain enough metadata to avoid reopening block files for ordinary indexed questions.

============================================================
13. WHY BLOCK METADATA IN LMDB IS NOT POINTLESS
============================================================

It is useful redundancy.

The raw block file is for:
- canonical full bytes
- append-only archival history

LMDB is for:
- indexed access
- fast metadata queries
- chain selection
- validation status
- relationship tracking
- operational behavior

So the split is:

Flat block files:
- the raw object

LMDB:
- how to find it
- how to classify it
- how to use it safely

That is the correct architecture.

============================================================
14. HOW IT WORKS IN PRACTICE
============================================================

Writing a new block:
--------------------

1. Fully validate the block
2. Append the raw block to the end of the current flat file
3. Record file number, byte offset, and byte length
4. Store block metadata in LMDB
5. Update height mapping in LMDB
6. Update chain tip and chainwork metadata in LMDB
7. Update UTXO state in LMDB
8. Update tx index if enabled
9. Remove mined mempool transactions if applicable

Reading a block by hash:
------------------------

1. Query LMDB block index by block hash
2. Retrieve:
   - file number
   - byte offset
   - byte length
3. Open the correct blkNNNN.dat file
4. Seek to the offset
5. Read the exact block bytes
6. Parse and return the block

Reading block metadata:
-----------------------

1. Query LMDB
2. Return metadata immediately
3. No need to reopen the block file unless full block bytes are actually needed

That is one of the biggest advantages.

============================================================
15. WHY THIS IS FAST
============================================================

This structure is fast because it matches the engine to the workload.

Flat files are best for:
- large raw block blobs
- append-only writes
- archival storage
- replay source of truth

LMDB is best for:
- random indexed lookup
- hot metadata access
- UTXO access
- tx index access
- chainstate updates
- validation status tracking

The real performance bottlenecks in a blockchain node are often:
- signature verification
- UTXO lookup/update
- mempool validation
- block validation
- propagation
- sync logic

The hybrid design keeps those hot paths on LMDB while keeping raw historical blocks in a storage format that is ideal for them.

============================================================
16. WHY NOT STORE FULL BLOCKS IN LMDB LONG TERM
============================================================

LMDB can store full blocks.
That is technically fine.
But it is less ideal long term because:

- raw block blobs are large
- archival history gets mixed with hot mutable state
- database growth becomes less elegant
- append-only history is not using the most natural storage model
- reindex/recovery becomes less clean
- hot-state and cold-history concerns become blurred

LMDB is still the right place for metadata.
It is just not the ideal long-term home for the full raw block bytes.

============================================================
17. WHY NOT STORE EVERYTHING IN BLOCK FILES
============================================================

Because block files are not a good engine for:
- UTXO lookup
- tx index lookup
- chain tip metadata
- validation status lookup
- side-chain/orphan tracking
- reorg handling metadata
- peer metadata
- hot random-access state

So Atho should NOT move UTXOs out of LMDB.
UTXOs should stay in LMDB.

============================================================
18. RECOVERY AND REINDEX BENEFITS
============================================================

One of the biggest advantages of flat block files is recovery.

If LMDB indexes or chainstate are damaged, Atho can:
- read raw blocks from blk*.dat
- rebuild block index
- rebuild metadata
- rebuild tx index
- rebuild chainstate / UTXO state

That is extremely valuable.

Flat block files become:
- the canonical archival source of truth for raw blocks

LMDB becomes:
- the fast working index/state layer

This is a very strong production design.

============================================================
19. RECOMMENDED ATHO LMDB GROUPS
============================================================

Suggested LMDB logical groupings:

1. block_index.lmdb
   - block hash -> file number, offset, length, height, status, chainwork, timestamp, flags
   - height -> block hash
   - side-chain/orphan metadata if needed

2. chainstate.lmdb
   - UTXO set
   - chain tip
   - best height
   - best chainwork
   - chainstate metadata

3. txindex.lmdb
   - txid -> block location / block hash / tx position

4. peers.lmdb
   - known peers
   - metadata
   - timestamps
   - scores if used

5. mempool.lmdb or memory-first mempool with optional persistence
   - pending tx metadata
   - relay metadata
   - validation metadata if persisted

6. validation_meta.lmdb if separated
   - block status
   - processing markers
   - ruleset/version metadata
   - optional safe persistent cache metadata

Exact file grouping can vary, but block metadata must remain indexed and quickly accessible.

============================================================
20. RECOMMENDED ATHO WRITE ORDER
============================================================

When Atho accepts a block, the safe conceptual order is:

1. fully validate block
2. append raw block to flat file
3. record storage location
4. write block metadata/index to LMDB
5. batch update UTXO state in LMDB
6. update chain tip metadata
7. update tx index if enabled
8. clear or update mempool state

Be careful about partial failure handling.

The implementation should ensure:
- no accepted block loses its location metadata
- no chain tip points to a block not safely recorded
- no partial updates leave corrupted visible state

============================================================
21. RECOMMENDED ATHO READ PATHS
============================================================

Block metadata query:
- LMDB only

Block by height:
- LMDB height map
- then LMDB block metadata
- then flat file if raw block bytes are needed

Block by hash:
- LMDB metadata
- then flat file if raw block bytes are needed

Chain tip query:
- LMDB only

UTXO query:
- LMDB only

This is why the hybrid design is so strong.
It keeps the common hot queries fast.

============================================================
22. FILE ROTATION AND MAINTENANCE
============================================================

Atho should not keep one endlessly growing block file.

Recommended:
- rotate block files at a fixed size threshold
- keep file naming deterministic
- keep file IDs stable
- keep index entries pointing to file id + offset + length

This makes:
- files easier to manage
- backups easier
- corruption isolation better
- storage organization cleaner

============================================================
23. REORG BEHAVIOR
============================================================

Reorgs do not require deleting raw block bytes from block files.

Instead:
- old branch blocks can remain stored
- LMDB metadata determines which branch is active
- chain tip / main-chain flags / chainwork decide the best chain
- chainstate in LMDB is rolled back and reapplied as needed

This is another reason metadata in LMDB is essential.
The flat files are not making branch decisions.
LMDB metadata and chainstate are.

============================================================
24. SUMMARY OF THE HYBRID APPROACH
============================================================

The hybrid approach means:

Flat files:
- store the raw full block bytes
- append-only
- archival
- source of truth for block payloads

LMDB:
- stores where the block is
- stores metadata about the block
- stores how the block fits into the chain
- stores chainstate and indexes
- stores everything hot and mutable

This is the correct long-term production design.

============================================================
25. FINAL ATHO DECISION
============================================================

For Atho:

Keep in LMDB:
- UTXO set
- block metadata
- block index

- chain metadata
- chain tip metadata
- peer metadata
- mempool metadata if persisted

Move to flat block files:

============================================================
26. FINAL SUMMARY
============================================================

Bitcoin's model works because it uses the right tool for each job.

Bitcoin-style flat block files:
- best for full raw blocks
- best for append-only history
- best for replay/reindex source-of-truth storage

LMDB:
- best for UTXOs
- best for indexes
- best for chain metadata
- best for validation metadata
- best for random lookup and mutable state

Atho should use this same hybrid structure because it is:
- faster
- cleaner
- easier to maintain
- easier to recover
- better long term
- more production-grade

Final recommendation:
---------------------
Atho should store full raw blocks in Bitcoin-style flat block files and keep block metadata plus everything else in LMDB.

That is the fastest serious long-term storage structure for Atho.
